**Findings**
1. 严重：这套方案把 CEO 变成了正常路由、异常治理、上线放量、恢复执行的统一同步闸门。人工 issue 先进入 `CEO intake`，system issue 也默认先进 `CEO intake`；critical block 之后，哪怕部门已修好、review 已过、canary 已成功，仍要 CEO 依次决定 canary、full live、resume、final close。对部长来说，这意味着“技术上已恢复”不等于“业务上可继续”；对人类操作员来说，这会把 MTTR 和日常分诊都堆到同一个队列里。定位：[部门模型 5.1](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L249) [部门模型 6.5-7.2](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L388) [执行插件 10.1-10.4](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-execution-improvement-plugin.md#L298) [热更新 7.2](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L264) [热更新 7.5](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L298) [热更新 7.8-8.1](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L330)

2. 严重：`shared system project` 被定义成“普通 project + 保留语义”，但实际承载的是第二套 issue 模型。文档一边说它保持 normal project portability / simpler UI，一边又给它单独的 issue type、状态机、排序、可见性、CEO-only closure、`ready_to_resume` 语义。当前核心 issue 只有一套状态机；如果 system issue 继续伪装成普通 issue，CEO/部长/人类会在同一个项目系统里遇到两套完全不同的规则，这是很难解释清楚的产品面。定位：[部门模型 6.1](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L331) [部门模型 6.4](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L369) [部门模型 7.1](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L421) [V1 Issue 状态机](/D:/Source/workspace/paperclip/doc/SPEC-implementation.md#L376) [执行插件 18.1](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-execution-improvement-plugin.md#L602)

3. 高：`Skills System Plugin` 会和现有 core `/skills` 直接形成双真相。方案明确把 “plugin-owned skill records / registry / candidates / promotion requests” 作为 MVP，同时又把“直接修改现有 company skill library 作为 source of truth”列为非目标；但当前产品已经有核心 `/skills/*` 页面、company skill API，以及 `sourceType / trustLevel / compatibility / usedByAgents` 这些一等模型。对 CEO/部长而言，这会变成两套技能中心并存，最容易失败的点是：推荐/候选在插件页，实际挂载/编辑/更新却在 core 页，谁才代表“已批准可用”并不清楚。定位：[技能插件 6.1-6.3](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-skills-system-plugin.md#L233) [技能插件 6.2](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-skills-system-plugin.md#L243) [主应用 `/skills` 路由](/D:/Source/workspace/paperclip/ui/src/App.tsx#L132) [company skill 路由](/D:/Source/workspace/paperclip/server/src/routes/company-skills.ts#L54) [company skill 核心类型](/D:/Source/workspace/paperclip/packages/shared/src/types/company-skill.ts#L14)

4. 高：通知与恢复动作被切到了 email，而不是留在控制面主流程里。critical block 和 `ready_to_resume` 都以 email 为 primary channel，且默认抄送 human operator；但产品定义强调 default UI 要回答 “what needs my approval”，工作也应附着在 work objects 上。现在的设计会把 CEO 的动作拆到 system issue、plugin page、邮箱三处，既容易制造噪音，也容易让真正该点的恢复动作藏在邮箱线程里。定位：[执行插件 13](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-execution-improvement-plugin.md#L422) [热更新 8](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L344) [PRODUCT: Board-level abstraction](/D:/Source/workspace/paperclip/doc/PRODUCT.md#L127)

5. 中高：两个 system plugin 被定义为实例级必装、默认启用、正常情况下不可停用，但它们最关键的 tuning knobs 又是 threshold / noise suppression / reminder timing 这类高影响参数。当前插件运行时明确是 instance-wide install，company route 只是 UI 上的上下文包装；这意味着多公司实例里，一个公司的噪音问题或阈值选择会影响所有公司，而且在“正常 operation”里还拿掉了最直接的止损手段。定位：[部门模型 10](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L616) [执行插件 3](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-execution-improvement-plugin.md#L53) [技能插件 2.1](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-skills-system-plugin.md#L59) [PLUGIN_SPEC: instance-level routes](/D:/Source/workspace/paperclip/doc/plugins/PLUGIN_SPEC.md#L952) [plugin SDK caveats](/D:/Source/workspace/paperclip/packages/plugins/sdk/README.md#L40) [PRODUCT: multi-company instance](/D:/Source/workspace/paperclip/doc/PRODUCT.md#L5)

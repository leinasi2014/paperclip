**Findings**

1. `严重` 部门 intake、系统治理、恢复授权这三套状态没有落到同一个可实现状态机上，而且文档之间已经出现语义冲突。部门文档把 `CEO intake`、`accept/reject/needs_clarification`、`pending_review/review_passed/ready_to_resume` 都当成流程状态使用，但同时又把“具体落在哪个数据库/UI 状态模型”留成 open question。更糟的是，执行改进文档写的是“review 成功后进入 `ready_to_resume`”，自演进文档写的是“review 成功后先 ready for rollout，完成 canary/full cutover 后才进入 `ready_to_resume`”。现有核心 `issues` 只有 `backlog|todo|in_progress|in_review|done|blocked|cancelled`，也没有 `departmentId`、`type`、`CEO intake` 或 review gate 字段；如果把这些全塞进现有 `status`，会直接冲撞当前 issue 路由/UI/状态守卫，如果不塞进去，则没有承载实体。参考：[部门文档 §5.1](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L249) [部门文档 §6.4](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L369) [部门文档 open question](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L721) [执行改进 §10.4](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-execution-improvement-plugin.md#L330) [自演进 §7.4](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L289) [当前 issue 常量](/D:/Source/workspace/paperclip/packages/shared/src/constants.ts#L112) [当前 issues schema](/D:/Source/workspace/paperclip/packages/db/src/schema/issues.ts#L21) [V1 spec issue 状态机](/D:/Source/workspace/paperclip/doc/SPEC-implementation.md#L376)

2. `严重` “临时工/子代理不是正式员工”这个前提，与当前完全 agent-centric 的执行/认证/计费模型正面冲突，但文档没有给出规范化落点。部门文档要求 temporary worker 有 owner、status、TTL、权限、pause/resume、issue assignment 和预算责任；但当前运行面所有关键外键都指向 `agents`：`issues.assigneeAgentId`、`heartbeat_runs.agentId`、`cost_events.agentId`、`agent_api_keys.agentId`，权限主体也只有 `user|agent`。如果 temporary worker 不是 `agents` 行，就需要复制整套 auth/run/cost/assignment 体系；如果它其实还是 `agents` 行，那它为什么不走当前 formal hire / `pending_approval` / org chart 逻辑，文档也没定义。现在提的 `ownerMinisterAgentId/sourceIssueId/ttl/scope/status` 元数据不够承接这套现有生命周期。参考：[部门文档 §8.1](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L482) [部门文档 §8.2](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L496) [当前 agents schema](/D:/Source/workspace/paperclip/packages/db/src/schema/agents.ts#L13) [当前 issues assignee](/D:/Source/workspace/paperclip/packages/db/src/schema/issues.ts#L34) [当前 heartbeat_runs](/D:/Source/workspace/paperclip/packages/db/src/schema/heartbeat_runs.ts#L6) [当前 cost_events](/D:/Source/workspace/paperclip/packages/db/src/schema/cost_events.ts#L9) [当前 agent_api_keys](/D:/Source/workspace/paperclip/packages/db/src/schema/agent_api_keys.ts#L5) [当前 principal types](/D:/Source/workspace/paperclip/packages/shared/src/constants.ts#L334) [当前 hire 流程进入 pending_approval](/D:/Source/workspace/paperclip/server/src/routes/agents.ts#L1286)

3. `高` Skills System Phase A 明确要做“plugin-owned registry”，同时又明确“不直接修改现有 company skill library 作为 source of truth”，这会立刻把技能系统做成双主库。当前仓库已经有核心 `company_skills` 表、完整 CRUD/import/update 路由、主应用 `/skills` 页面，而且 agent skill sync 也是通过 `companySkillService` 解析和下发 runtime skills；公司 portability 也已经把 skills 纳入导入导出契约。也就是说，Phase A 如果只写 `plugin_entities/plugin_state`，那 plugin UI 看到的 skill、agent 实际可同步的 skill、以及导出的 skill 会分裂成三套视图。这个不是后续优化问题，而是 MVP 就会出现的 source-of-truth 冲突。参考：[技能插件决策摘要](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-skills-system-plugin.md#L42) [技能插件 MVP goals](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-skills-system-plugin.md#L233) [技能插件 MVP non-goals](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-skills-system-plugin.md#L243) [技能插件数据模型](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-skills-system-plugin.md#L272) [当前 company_skills schema](/D:/Source/workspace/paperclip/packages/db/src/schema/company_skills.ts#L12) [当前 company skill routes](/D:/Source/workspace/paperclip/server/src/routes/company-skills.ts#L54) [当前 agent skill sync](/D:/Source/workspace/paperclip/server/src/routes/agents.ts#L740) [当前主应用 skills 路由](/D:/Source/workspace/paperclip/ui/src/App.tsx#L132) [当前 portability 已含 skills](/D:/Source/workspace/paperclip/doc/SPEC-implementation.md#L843)

4. `高` 部门预算 envelope、预留/结算、部门级 blocking 在现有预算模型里没有承载位置。部门文档要求 `monthlyLimit/reserved/spent`、`allocated/reserved_only/active`、执行前 reserve、执行后 settle，并把 temporary worker spend 归集到 department；执行改进文档又要求 issue-chain blocking 和 department-level blocking。但当前共享常量只支持 `company|agent|project` 三种 budget scope，`budget_policies/budget_incidents` 只是阈值策略和告警事件，`cost_events` 只记录实际发生的成本，没有 reservation ledger，也没有“部门冻结/部门恢复/链路级 block”这种可查询对象。换句话说，这里缺的不是 UI，而是核心账本和状态对象。参考：[部门文档 §4.2](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L182) [部门文档 §4.3](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L211) [部门文档 §4.4](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L231) [执行改进 §10.3](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-execution-improvement-plugin.md#L322) [当前 budget scope](/D:/Source/workspace/paperclip/packages/shared/src/constants.ts#L267) [当前 budget_policies](/D:/Source/workspace/paperclip/packages/db/src/schema/budget_policies.ts#L4) [当前 budget_incidents](/D:/Source/workspace/paperclip/packages/db/src/schema/budget_incidents.ts#L7) [当前 cost_events](/D:/Source/workspace/paperclip/packages/db/src/schema/cost_events.ts#L9)

5. `中` Hot-update 方案定义了 live/candidate、slot A/B、canary/full、drain、rollback，但没有给出持久化 rollout 对象，现有 plugin 持久层也装不下这个模型。当前 `plugins` 是“一插件一行，一个 version，一个 status”，`plugin_state` 是当前态 KV，`plugin_entities` 是插件自定义记录；它们都不能稳定表示“当前 live 是谁、candidate 是谁、谁批准了 canary、哪次 rollout 失败回滚、哪次切流完成排空”。而文档同时要求这是一个 governed、auditable 的 core-owned rollout pipeline。没有 first-class rollout/deployment record，控制器无法做到可恢复和可审计，尤其是在“更新正在负责诊断的插件自己”这种场景。参考：[自演进文档 §3.5](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L106) [自演进文档 §5.1](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L171) [自演进文档 §7.6](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L308) [自演进文档 §11.2](/D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L431) [当前 plugins schema](/D:/Source/workspace/paperclip/packages/db/src/schema/plugins.ts#L23) [当前 plugin_state schema](/D:/Source/workspace/paperclip/packages/db/src/schema/plugin_state.ts#L13) [当前 plugin_entities schema](/D:/Source/workspace/paperclip/packages/db/src/schema/plugin_entities.ts#L13)

**Findings**

1. 严重：双槽热更新设计与当前插件运行时模型不兼容。  
   定位：`Self-Evolution and Hot Update` 的双槽/灰度部分，[§5.2-§6]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L184 )，[§5.3]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L195 )，[§5.4]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L204 )，[§5.5]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L210 )，[§6]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L229 )，[§7.6-§7.8]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L310 )；同文又把 “candidate-versus-live evaluation” 列为“无需 core changes 可先发”，见 [§10]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L414 )，但后面又承认需要 `dual-slot plugin lifecycle` 和 `canary routing`，见 [§11.2]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L435 )。  
   现状：当前运行时只有“单 pluginId = 单 worker/单状态/单配置/单状态空间/单 job 命名空间”。证据见 [plugin-worker-manager.ts#L6]( /D:/Source/workspace/paperclip/server/src/services/plugin-worker-manager.ts#L6 )、[plugin-worker-manager.ts#L1222]( /D:/Source/workspace/paperclip/server/src/services/plugin-worker-manager.ts#L1222 )、[plugin_config.ts#L19]( /D:/Source/workspace/paperclip/packages/db/src/schema/plugin_config.ts#L19 )、[plugin_config.ts#L28]( /D:/Source/workspace/paperclip/packages/db/src/schema/plugin_config.ts#L28 )、[plugin_state.ts#L17]( /D:/Source/workspace/paperclip/packages/db/src/schema/plugin_state.ts#L17 )、[plugin_state.ts#L75]( /D:/Source/workspace/paperclip/packages/db/src/schema/plugin_state.ts#L75 )、[plugin_jobs.ts#L35]( /D:/Source/workspace/paperclip/packages/db/src/schema/plugin_jobs.ts#L35 )、[plugin_jobs.ts#L54]( /D:/Source/workspace/paperclip/packages/db/src/schema/plugin_jobs.ts#L54 )、[plugin_ui_static.ts#L267]( /D:/Source/workspace/paperclip/server/src/routes/plugin-ui-static.ts#L267 )。  
   结论：按现状无法实现真正的 `live/candidate` 并行、`shadow/canary/full` 路由、旧版本继续跑新版本接新流量。直接照文档推进，只会把 live 和 candidate 写进同一份 `plugin_state/config/jobs`，造成状态漂移、回滚不可信。

2. 严重：文档把“system plugin 只诊断/建议、不直接治理”当成职责边界，但当前宿主没有把这条边界做成技术边界。  
   定位：`Execution Improvement Plugin` 的边界声明，[§5]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-execution-improvement-plugin.md#L119 )、[§10]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-execution-improvement-plugin.md#L305 )、[§12.2]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-execution-improvement-plugin.md#L403 )；`Skills System Plugin` 的边界声明，[§4.3]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-skills-system-plugin.md#L152 )。  
   现状：插件 worker 只要声明能力，就能直接 `issues.update`、`agents.pause/resume/invoke`，见 [host-client-factory.ts#L305]( /D:/Source/workspace/paperclip/packages/plugins/sdk/src/host-client-factory.ts#L305 )、[host-client-factory.ts#L318]( /D:/Source/workspace/paperclip/packages/plugins/sdk/src/host-client-factory.ts#L318 )、[host-client-factory.ts#L490]( /D:/Source/workspace/paperclip/packages/plugins/sdk/src/host-client-factory.ts#L490 )、[host-client-factory.ts#L521]( /D:/Source/workspace/paperclip/packages/plugins/sdk/src/host-client-factory.ts#L521 )、[host-client-factory.ts#L524]( /D:/Source/workspace/paperclip/packages/plugins/sdk/src/host-client-factory.ts#L524 )、[host-client-factory.ts#L527]( /D:/Source/workspace/paperclip/packages/plugins/sdk/src/host-client-factory.ts#L527 )，宿主实现也确实把这些能力暴露出去了，见 [plugin-host-services.ts#L779]( /D:/Source/workspace/paperclip/server/src/services/plugin-host-services.ts#L779 )、[plugin-host-services.ts#L857]( /D:/Source/workspace/paperclip/server/src/services/plugin-host-services.ts#L857 )、[plugin-host-services.ts#L864]( /D:/Source/workspace/paperclip/server/src/services/plugin-host-services.ts#L864 )、[plugin-host-services.ts#L871]( /D:/Source/workspace/paperclip/server/src/services/plugin-host-services.ts#L871 )。更关键的是，插件 UI 仍是同源可信 JS，manifest capability 并不约束它调用普通 HTTP API，见 [PLUGIN_SPEC.md#L23]( /D:/Source/workspace/paperclip/doc/plugins/PLUGIN_SPEC.md#L23 )、[PLUGIN_SPEC.md#L24]( /D:/Source/workspace/paperclip/doc/plugins/PLUGIN_SPEC.md#L24 )、[PLUGIN_AUTHORING_GUIDE.md#L9]( /D:/Source/workspace/paperclip/doc/plugins/PLUGIN_AUTHORING_GUIDE.md#L9 )、[PLUGIN_AUTHORING_GUIDE.md#L12]( /D:/Source/workspace/paperclip/doc/plugins/PLUGIN_AUTHORING_GUIDE.md#L12 )。  
   结论：文档里“插件不能直接关单/不能成为 shadow control plane/只有 CEO 能 resume”目前只是约定，不是强制约束。对 system plugin 来说，这会直接把 core governance 变成可绕过的软边界。

3. 高：`required system plugins` 的设计与现有插件合同和宿主管理面不兼容。  
   定位：`Department-Minister and Subagent Organization Model` 的 [§10]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L616 )，尤其 [L620]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L620 )、[L632]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-department-minister-and-subagent-org-model.md#L632 )；`Execution Improvement Plugin` 的安装模型，[L55]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-execution-improvement-plugin.md#L55 )、[L61]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-execution-improvement-plugin.md#L61 )、[L71]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-execution-improvement-plugin.md#L71 )；`Skills System Plugin` 的安装模型，[L61]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-skills-system-plugin.md#L61 )、[L67]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-skills-system-plugin.md#L67 )、[L77]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-skills-system-plugin.md#L77 )。  
   现状：当前 manifest 和持久化模型里没有 `system|required|nonDisableable` 这样的生命周期字段，见 [plugin.ts#L201]( /D:/Source/workspace/paperclip/packages/shared/src/types/plugin.ts#L201 )、[plugin.ts#L260]( /D:/Source/workspace/paperclip/packages/shared/src/types/plugin.ts#L260 )；而插件路由默认对所有插件开放启用/禁用/卸载，见 [plugins.ts#L1230]( /D:/Source/workspace/paperclip/server/src/routes/plugins.ts#L1230 )、[plugins.ts#L1266]( /D:/Source/workspace/paperclip/server/src/routes/plugins.ts#L1266 )、[plugins.ts#L1304]( /D:/Source/workspace/paperclip/server/src/routes/plugins.ts#L1304 )。  
   结论：文档把这两个插件定义成“可见但不可移除/不可禁用”的系统基础设施，但当前宿主仍把它们当普通插件处理。没有单独的 core 级生命周期类型之前，这个设计会把系统关键能力暴露为普通运维按钮。

4. 高：`不打断正常运行`、`自动回滚` 的表述显著高估了当前生命周期原语。  
   定位：`System Self-Evolution and Hot Update` 的总体承诺，[L23]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L23 )，以及 [§5.3-§5.5]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L195 )、[§6]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L229 )、[§7.6-§7.8]( /D:/Source/workspace/paperclip/doc/plans/2026-04-02-system-self-evolution-and-hot-update.md#L310 )。  
   现状：当前“热变更”本质还是 restart/replace。配置变更如果插件不处理 `configChanged`，宿主直接 `restartWorker`，见 [plugins.ts#L1593]( /D:/Source/workspace/paperclip/server/src/routes/plugins.ts#L1593 )、[plugins.ts#L1610]( /D:/Source/workspace/paperclip/server/src/routes/plugins.ts#L1610 )；worker 停机流程是 `shutdown RPC -> SIGTERM -> SIGKILL`，见 [plugin-worker-manager.ts#L66]( /D:/Source/workspace/paperclip/server/src/services/plugin-worker-manager.ts#L66 )、[plugin-worker-manager.ts#L874]( /D:/Source/workspace/paperclip/server/src/services/plugin-worker-manager.ts#L874 )、[plugin-worker-manager.ts#L900]( /D:/Source/workspace/paperclip/server/src/services/plugin-worker-manager.ts#L900 )、[plugin-worker-manager.ts#L909]( /D:/Source/workspace/paperclip/server/src/services/plugin-worker-manager.ts#L909 )；job 注销会把在途 run 标成 `cancelled`，见 [plugin-job-scheduler.ts#L642]( /D:/Source/workspace/paperclip/server/src/services/plugin-job-scheduler.ts#L642 )、[plugin-job-scheduler.ts#L662]( /D:/Source/workspace/paperclip/server/src/services/plugin-job-scheduler.ts#L662 )；开发态文件变更同样是直接重启 worker，见 [plugin-dev-watcher.ts#L219]( /D:/Source/workspace/paperclip/server/src/services/plugin-dev-watcher.ts#L219 )、[plugin-dev-watcher.ts#L222]( /D:/Source/workspace/paperclip/server/src/services/plugin-dev-watcher.ts#L222 )。另外，连 crash/restart 清理链路都还比较脆弱：宿主创建了 crash cleanup 控制器，但 `workerManager` 并没有把 `onWorkerEvent` 接上，见 [app.ts#L159]( /D:/Source/workspace/paperclip/server/src/app.ts#L159 )、[app.ts#L181]( /D:/Source/workspace/paperclip/server/src/app.ts#L181 )、[plugin-host-service-cleanup.ts#L41]( /D:/Source/workspace/paperclip/server/src/services/plugin-host-service-cleanup.ts#L41 )、[plugin-worker-manager.ts#L1183]( /D:/Source/workspace/paperclip/server/src/services/plugin-worker-manager.ts#L1183 )；而 `dispose()` 本身还明确写着“不清理会在 crash/restart 后产生重复订阅”，见 [plugin-host-services.ts#L1113]( /D:/Source/workspace/paperclip/server/src/services/plugin-host-services.ts#L1113 )。  
   结论：在现有原语上，系统可以做到“热重启式更新”，但还远不到文档描述的“无中断 hot switch + 可信 rollback”。如果把 system plugin 放进这套链路，停机、状态漂移、重复订阅、job 取消都是真实风险。

**Open Questions**

- `Execution Improvement Plugin` 所说的“immediately block the relevant execution path”在 core 里究竟对应什么对象？是 `issue.status=blocked`、`agent.pause`、heartbeat wakeup policy，还是一个新的 `execution_block`/`resume_gate` 实体？
- 这两个 system plugin 真的还要走普通插件生命周期吗？如果答案是“要”，那 `required/non-disableable/non-removable` 的宿主字段、校验和路由豁免应该落在哪一层？
- 如果继续推进 `live/candidate`，隔离键是什么？`plugin_id + slot`、`plugin_release_id`，还是别的 release 对象？`plugin_state`、`plugin_config`、`plugin_jobs`、`plugin_logs` 需要按哪个维度拆分？
- 热更新范围是否包含 plugin UI bundle？如果包含，当前同源 UI 的缓存、已挂载组件树、bridge 上下文和 query 生命周期该如何安全切换？
- 这些方案是否明确只面向当前“单节点、持久化本地文件系统”的部署模型？如果不是，artifact 分发、跨节点 canary、回滚一致性由谁负责？
